Fuzzing and Mozilla: 2016

Gary Kwong, @nth10sd or gkw@mozilla.com

This presentation describes where fuzzing sits in a Mozilla development workflow in 2016. It will mention how you can leverage it in your workflow to make development more efficient, and will not cover fuzzing itself in detail.

English version by Gary Kwong.


Abstract: The Mozilla Gecko platform has many moving parts. There are increasingly more areas which need testing to ensure good quality and safety of the users. In this presentation, Senior Security Engineer Gary Kwong will describe how fuzzing can be leveraged in the Mozilla development workflow. Gary will talk about fuzzing tools, suggested automation building blocks and ways to start deploying fuzzing quickly. A case study on the success of the fuzzing team working together with the JavaScript team will be described. Furthermore, he will also cover the potential pitfalls of this approach and conditions under which fuzzing can be most effective.

What is fuzzing?

  • Give random input to software
  • May find easy-to-detect problems
    • Crashes
    • Assertion failures
      • 1194022
  • These can manifest in the real-world
Fuzzing can also discover races. Fuzzbugs can be rare corner cases, and can also appear in the real-world.

Dynamic analysis tools

  1. Address Sanitizer (ASan)
  2. Thread Sanitizer (TSan)
  3. Leak Sanitizer (LSan)
  4. Valgrind
  5. and Undefined Behaviour Sanitizer (UBSan), etc.
These tools help find memory safety problems, thread races, use-after-frees and buffer overflows.

Possible focus areas for sec bugs

  • Modules that:
    • Are written in a unsafe language
    • Are less sandboxed
    • Parse attacker-controlled input
    • Implement security boundaries
    • Maintain critical invariants
    • Other security researchers are working on
Security bugs tend to also show up for the user.

Working as a team

  • More starting points for fuzzing

Devs code to help find more bugs

  • Add assertions & testing functions
  • Test components in isolation
  • Dynamic analysis tool support
    • ASan
    • TSan
    • Valgrind
Test components in isolation: Similar to SpiderMonkey shell but for mobile radio and USB; Dynamic analysis tool support: Run testsuites with dynamic analysis tools ideally before fuzzing

Results

  • Security bugs → higher-priority
  • Fuzzblockers → higher-priority
  • Fuzzbugs → generally lower-priority
  • The more we discover, the less others find
  • More bugs discovered, as we scale
Security bugs, fuzzblockers → higher-priority; Fuzzbugs → generally lower-priority； Security researchers find less issues that make us scramble to chem-spill, and we get better bug reports from fuzzing.

Potential pitfalls

  • Automation frameworks
  • Devices not powerful enough
Lack of stacktraces when Gaia apps crash; Lack ways to collect kernel failures; Automation frameworks need sufficient developer attention; Devices not powerful enough for dynamic analysis tools

Factors for success

  • Patches land on the trunk (master) first
  • Backout when regressions occur
  • Fuzzblockers are fixed
  • Tests are rarely disabled

Suggested automation building blocks

  • Input creation (the fuzzer)
  • Instrumentation
  • Monitoring
  • Log bucketing
  • Testcase reduction
  • Bisection
The workflow start should result in a workflow ending consisting of reduced testcases with bisection results. Results can also end up in a dashboard.
